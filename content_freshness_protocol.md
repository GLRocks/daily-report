## 1. 搜索层强制规则（Search Enforcement）

### 1.1 时间筛选强制
- **每条搜索必须使用 `freshness=day` 或 `date_after: today-2`**
- 禁止使用无时间筛选的通用搜索（如直接搜索 "OpenAI GPT"）
- 正确示例: `OpenAI GPT latest release freshness:day` 或 `date_after:2026-05-10`
- 错误示例: `OpenAI GPT news`（无时间筛选 = 可能返回过时结果）

### 1.2 来源层级强制
- **Tier 1来源优先**：官方博客、SEC文件、 earnings call transcript、技术论文
- **Tier 2来源验证**：Reuters/Bloomberg/FT原创报道、DigiTimes/EETimes（具名信源）
- **禁止来源**：36kr/钛媒体/品玩/雷锋网、Reddit/Twitter threads作主源、"据知情人士"无署名

### 1.3 交叉验证强制
- **每个关键事实必须来自≥2个独立来源交叉验证**
- 独立来源定义：不同机构/不同记者/不同语言区报道
- 必须验证的关键事实类型：
  - 模型版本号与发布日期
  - 财报数据（收入/利润/growth rate）
  - 产品定价与规格
  - 人物言论（CEO interview、conference keynote）
  - 政策/法规生效日期

## 2. 事实验证层（Fact Verification）

### 2.1 模型信息验证
| 字段 | 验证规则 | 失败处理 |
|------|----------|----------|
| 模型版本号 | 与官方GitHub/blog/announcement核对 | 标注"版本号待确认"或不收录 |
| 发布日期 | 与官方发布渠道交叉验证 | 标注"日期待确认"或不收录 |
| 上下文窗口 | 与官方技术报告核对 | 不收录 |
| 定价 | 与官方pricing page核对 | 不收录 |
| 开源策略 | 与GitHub license核对 | 不收录 |

### 2.2 财报数据验证
| 字段 | 验证规则 | 失败处理 |
|------|----------|----------|
| 营收 | 与SEC filing/earnings call核对 | 标注"Q{季度}数据"并确认是否最新 |
| 毛利率 | 同上 | 同上 |
| 增长率 | 同上 | 同上 |
| capex指引 | 与法说会transcript核对 | 同上 |

### 2.3 股价/价格数据验证
| 字段 | 验证规则 | 失败处理 |
|------|----------|----------|
| 股票价格 | 100%来自ifind API `realtime_price`的`close`字段 | API失败→标注"价格暂缺"，禁止搜索回填 |
| 日内涨跌 | 使用API的`open`和`close`计算：`(close-open)/open×100%` | 不使用`pct_change_1m`（1分钟无意义） |
| 数据日期 | 必须标注`YYYY-MM-DD` | 无日期→不收录 |
| YTD | 禁止硬编码；如需显示必须搜索获取+标注来源+交叉验证 | 无法验证→不显示 |

**股票数据红线**：
- ❌ 禁止搜索引擎回填股价（INTC $24.50错误即源于此——API失败时搜索返回了2024年底旧价格）
- ❌ 禁止直接使用`pct_change_1m`作为日涨跌幅（该字段是1分钟变动，≈0%，非日涨跌）
- ❌ 禁止在stock card中硬编码YTD数字（除非通过可靠来源验证）
- ✅ 必须：API获取全部20只→检查close有效性→写入CSV→模板从CSV读取

### 2.4 人物言论验证
- **必须附原始来源链接**（YouTube/官方博客/Bloomberg interview）
- **必须标注发言日期和场合**
- **必须附上下文说明**（为什么此时发表此言论、与商业利益的关系）
- 禁止使用二手聚合网站的引用（如 "据报道，Jensen Huang说..."）

## 3. 去重与过时拦截（Deduplication & Stale Content Blocker）

### 3.1 近7日Topic追踪日志
- **文件**: `/root/.openclaw/workspace/memory/daily_topics_log.md`
- **格式**: 每行一条，格式 `YYYY-MM-DD | 板块 | Topic关键词 | 来源 | 投资含义摘要`
- **维护**: 每日生成晨报后自动追加当日覆盖的topic
- **使用**: 生成新晨报前读取此日志，避免重复覆盖近7日已讨论topic

### 3.2 重复拦截规则
- **近3日内已覆盖的Topic → 完全不收录**
- **近4-7日已覆盖的Topic → 仅当有新进展时以"跟进"形式收录**
- **新进展定义**: 有新数据/新事件/新评论，而非同一事件的不同角度报道

### 3.3 过时信息自动拦截
以下类型的信息**必须**标注日期并判断时效性：
- 产品发布信息 → 超过30天视为"背景信息"而非"新闻"，需有新进展才能收录
- 财报数据 → 必须使用最新已发布季度，禁止回溯上一季度数据填充
- 模型发布 → 超过14天的模型发布信息视为"已知信息"，需有更新版本/新功能/新数据才能收录
- 政策/法规 → 必须使用最新生效或最新变动信息，禁止重复报道已生效政策

## 4. 推送前自检清单（Pre-publish Checklist）

每日晨报生成后、推送前，必须运行以下自检：

### 4.1 版本号检查
- [ ] 所有AI模型版本号是否与官方最新一致？
- [ ] 是否有模型已发布新版本但推送中仍引用旧版本？

### 4.2 数据时效检查
- [ ] 所有财报数据是否为最新季度？
- [ ] 所有股价/价格数据是否标注日期？
- [ ] 所有政策/法规信息是否为最新变动？

### 4.3 去重检查
- [ ] 是否有topic在近7日日志中已出现？
- [ ] 如有重复，是否为新进展而非重复报道？

### 4.4 来源检查
- [ ] 是否有来自36kr/钛媒体/未署名"据知情人士"的信息？
- [ ] 国际观点占比是否≥80%？
- [ ] 每条信息是否有具体来源（机构/人名+日期）？

### 4.5 投资含义检查
- [ ] 每条信息是否通过"so what"测试？
- [ ] 是否有信息缺乏明确投资含义？

### 4.6 问责机制
- **如果自检不通过**: 标注"当日无重大投资信号"或"部分板块因信息不足暂缓推送"，**宁可不推也不推过时信息**
- **如果用户指出某条信息过时**: 记录到 `/root/.openclaw/workspace/memory/accuracy_errors.md`，追溯原因并修正规则
